TOPCIT 기술영역 소프트웨어 재사용

소프트웨어 재사용이란?

소프트웨어 개발관련 지식을 표준화하여 개발생산성을 높이기 위하여 반복적으로 사용하기에 적합하도록 구상하는 방법


소프트웨어 재사용의 원칙

  • 범용성 : 특정 응용분야만이 아닌 일반적으로 활용될 수 있는 정도이어야 한다.
  • 모듈성 : 정보은닉과 추상화의 원칙으로 최소한의 겨합도 및 최대한의 응집력을 갖도록 하는 특성이 있어야 한다.
  • 하드웨어 독립성 : 가능한 실행 하드웨어 기종과 무관해야 한다.
  • 소프트웨어 독립성 : OS(운영체제) 또는 DBMS(데이터베이스 관리 시스템)와는 무관하게 운영해야 한다.
  • 자기 문서화 : 모듈의 정확한 기능, 용법, 인터페이스를 기술한다.
  • 일반성 : 많은 개발자들에게 공통적으로 필요하고 사용 가능해야 한다.
  • 신뢰성: 품질을 믿고 사용할 수 있어야 한다.

실무에서 재사용 구현의 문제점

  • 공통으로 사용할 수 있는 소프트웨어 모듈을 발견하기가 어렵다.
  • 소프트웨어의 표준화가 부족하다.
  • 재사용을 위한 소프트웨어 부품은 개발비가 더 들 수 있다.

소프트웨어 재사용의 효과

  • 소프트웨어 생산의 TCO(총소유 비용) 절감
  • 높은 품질의 소프트웨어 생산을 위한 공유 및 활용효과
  • 시스템 개발에 대한 정보공유 및 타 프로젝트의 산출물 공유
  • 시스템 구조와 좋은 시스템 구축방법에 대한 교육적 효과

역공학(리버스 엔지니어링)이란?

이미 만들어진 시스템을 역으로 추적하여 처음의 문서나 설계기법 등의 자료를 얻어내는 일을 말한다.


역공학이 필요한 경우

  • 기 가동중인 시스템의 요지보수가 어려운 경우
  • 변경이 빈번하여 시스템 효율이 저하된 경우
  • 파일 시스템으로 개발된 업무를 관계형 데이터베이스로 재구축 하려는 경우
  • 기본 메인 프레임을 다운사이징 하는 경우

역공학의 종류

  • 논리역공학 : 물리적 설계정보를 얻어내는 역할 수행
  • 자료역공학 : 기존 데이터베이스를 수정하거나 새로운 데이터베이스 관리시스템으로 전이하는 역할수행

소프트웨어 설계 원리와 구조적 설계

소프트웨어 설계 원리

  • 추상화 : 상위 수준에서 제품의 구현을 먼저 생각하는 것으로 필수적인 것만 표현한다.
    컴포넌트의 구현 방법을 과감히 생략하고 외부 인터페이스에 초점을 맞추는 것이 추상화 개념이다.
  • 정보은닉 : 모듈의 내부 내용을 감추고 인터페이스를 통해서만 메시지를 전달할 수 있도록 한다.
    모듈 또는 하부 시스템이 다른 모듈의 구현에 영향을 받지 않게 설계되는 것을 의미한다.
  • 단게적 정제 : 프로그램 의 구조에서 점차 모듈에 대한 세부 사항으로 내려가며 구체화한다.
    정제 과정에서 추상화의 수준은 낮아지며 각 기능은 분해되어 해결방안을 제시한다.
  • 모듈화 : 시스템을 지능적으로 관리할 수 있도록 해 주며, 복잡도의 문제를 해결해 준다.
    크고 복잡한 문제를 해결하기 위해 문제를 작은 단위인 모듈로 분할 정복한다.
  • 구조화 : 시스템 구조화는 분할 과정에 의해 얻어진다.
    분할 과정은 앞의 요구사항 분석 과정에서 일차로 이루어지며, 설계 단계에서 더욱 세분화되어 나타나게 된다.

응집도와 결합도

좋은 설계란 설계 문서가 읽기 쉽고 이해하기 쉽게 만들어져야 하며, 시스템에 변화가 주어졌을 때 그 영향이 국소화되어야 한다.

  • 응집도 : 모듈 내의 각 구성 요소들이 공통의 목적을 달성하기 위하여 서로 얼마나 관련이 있는지 연관 정도를 나타낸다.
    모듈의 응집도를 높으면 낮은 결합도를 얻을 수 있다.
    • 기능적 응집도 : 모든 기능 요소들이 단일 문제와 연관되어 수행될 경우의 응집도
    • 순차적 응집도 : 하나의 활동으로부터 나온 출력 데이터를 그 다음 활동의 입력 데이터로 사용할 경우의 응집도
    • 교환적 응집도 : 동일한 입력과 출력 데이터를 그 다음 활동의 입력 데이터로 사용할 경우의 응집도
    • 절차적 응집도 : 구성 요소들이 기능을 순차적으로 수행할 경우의 응집도
    • 시간적 응집도 : 특정 시간에 처리되는 몇 개의 기능을 모아 하나의 모듈로 작성할 경우의 응집도
    • 논리적 응집도 : 유사한 성격을 갖거나 특정 형태로 분류되는 처리 요소들로 하나의 모듈이 형성되는 경우의 응집도
    • 우연적 응집도 : 구성요소들이 서로 관련 없는 요소로만 구성된 경우의 응집도
  • 결합도 : 모듈 사이의 상호 연관성의 복잡도를 일컫는다.
    결합도가 높을 수록 한 모듈을 독립적으로 변경하기 어렵고, 한 모듈의 변경이 다른 모듈에 미치는 영향도가 클 것으로 파문 효과를 일으킨다. 따라서 설게 시 최대한 결합도를 낮추는 것이 필요하다.
    • 자료 결합도 : 모듈 간의 내용을 전혀 알 필요가 없는 상태로 내용을 변경하더라도 다른 모듈에는 여향이 없는 결합도
    • 스탬프 결합도 : 배열이나 레코드 등의 자료 구조가 전달될 때의 결합도
    • 제어 결합도 : 논리적인 흐름을 제어하는 데 사용하는 제어 요소가 전달될 때의 결합도
    • 외부 결합도 : 외부로 선언한 데이터를 다른 모듈에서 참조할 때의 결합도
    • 공통 결합도 : 공통 데이터 영역을 여러 모듈이 사용할 때의 결합도
    • 내용 결합도 : 한 모듈이 다른 모듈의 내부 기능 및 그 내부 자료를 직접 참조하거나 수정할 때의 결합도

구조적 설계 방법

  • 변환흐름 중심 설계 : 정보를 받아들여 가공 처리한 후 그 결과인 외부세계에 출력하는 시스템을 이에 합당한 컴퓨터 구조로 매핑시키는 기법이다.
  • 트랜잭션흐름 중심 설계 : 들어온 입력을 여러 갈래의 출력흐름으로 쪼갤 수 있는 경우에 가능하다. 입력값을 평가하고 그 겨로가에 따라 여러 출력 경로 중 하나를 따라 흘러간다.
    트랜잭션 중심은 트랜잭션을 입력으로 받아들여 여러 출력 버블로 연결되어 있는 모습을 한 버블이다.

TOPCIT 기술영역 프로그래밍 언어와 개발환경

프로그래밍 언어의 개요

  • 프로그래밍 언어 개념
    • 저급언어 : 기계 중심적인 언어로 실행 속도가 빠르고 기계마다 다른 코드를 가짐
    • 고급언어 : 사람이 이해하기 위운 자연어에 가깝게 만들어짐
  • 인터프리터 언어 : 중간과정 없이 원시 프로그램을 직접 저급언어로 바꾸면서 동시에 실행
    • 장점 : 완전한 기계어 번역을 기다리지 않고 필요 시 마다 실행, 원시 언어 형태를 기억하므로 메모리 절약
    • 단점 : 원시 언어 형태를 유지하므로 메모리 절약
  • 컴파일러 언어 : 어느 특정 언어를 동등한 의미를 갖는 다른 언어로 바꾸는 것을 의미
    • 장점 : 번역된 코드 저장 가능, 컴파일 한 후 바로 재실행 가능
    • 단점 : 기계어로 변환하는데 많은 시간 소요, 메모리 낭비 발생 가능

주요 프로그래밍 언어의 특징

  • C언어 : 실행속도가 빠르고 효율적인 메모리 관리 용이, 절차지향적(프로그램 실행이 정해진 순서에 따라 진행)
  • C++언어 : C언어에서 객체 지향 프로그래밍을 지원하는 언어로, 캡슐화와 자료 은닉, 상속성과 재사용성, 다형성이 강조됨
  • Java언어 : C++의 복잡성을 단순화, 항상 장동 가비지 콜렉션을 수행
  • Python언어 : 동적 타이핑으로 실행 시간에 자료형을 확인가능, 멀티패러다임 프로그래밍 언어(절차적, 함수형, 객체 지향)
  • JavaScript언어 : 간단하게 코딩을 짜고 프로그래밍 가능, 별도의 컴파일 과정이 필요하지 않아 쉽게 배울 수 있음

소프트웨어 개발 프레임워크

  • 소프트웨어 개발 프레임워크 : 효율적인 개발 정보시스템 개발을 위한 코드 라이브러리, 인터페이스 규약, 설정정보 등의 집합
    이점 : 코드 품질 향상, 개발 생산성 증대, 요지보수성 향상, 신규시스템 위험 감소
  • 스프링 프레임워크 : 자바 플랫폼을 위한 오픈소스 어플리케이션 프레임워크, 안정성과 유연성이 입증됨
  • 전자정부 표주프레임워크 : 시스템 개발, 운영 시 필요한 기본 기능들을 표준화하여 미리 구현해 둔 공공사업용 표준개발프레임워크
    특징 : 개방형 표준준수, 사용 솔루션 연계, 국가적 표준화 지향, 변화 유연성
    구성요소 : 실행환경, 개발환경, 운영환경, 관리환경, 공통컴포넌트

통합개발환경(IDE)

  • 통합개발환경 : 코딩, 디버깅, 컴파일, 배포 등 프로그램 개발에 관련된 모든 작업을 하나의 소프트웨어 안에서 처리하는 환경을 제공하는 개발용 소프트웨어
    구성요소 : 편집기, 빌드도구, 디버거, 프로젝트 관리
  • CI(Continuous Integration) : 여러 명으로 구성된 팀이 작업한 결과를 자주 통합하는 소프트웨어 개발 활동
  • 소프트웨어 빌드 : 소스 코드 파일을 컴퓨터에서 실행할 수 있는 독립 소프트웨어 가공물로 변환하는 가정
  • 일일 빌드 : 매일 전체적으로 다시 컴파일하고, 기본 동작을 검증하기 위해 필요한 일련의 테스트를 거치는 공정
  • 소프트웨어 배포 : 소프트웨어 시스템을 만드는 모든 행위를 지칭

소프트웨어 테스팅과 리팩토링

테스팅이란?

어플리케이션 또는 시스템의 동작과 성능, 안정성이 사용자나 고객이 요구하는 수준을 만족하는지 확인이나 검증하기 위해 결함을 발견하는 방법


테스팅의 일반적인 원리

  • 결함의 존재성을 밝히는 활동
  • 완벽한 테스팅은 불가능
  • 테스팅은 개발 초기에 시작
  • 살충제 패러독스 : 테스트가 반복적으로 수행되면 더 이상 새로운 버그를 찾아내지 못함
  • 테스팅은 정황에 의존적
  • 오류-부재의 궤변

테스팅 프로세스

테스팅은 다양한 구성요스를 포함하는 프로세스를 중심으로 조정하고 관리되어야 한다.
테스팅 프로세스의 주요활동에는 테스트 계획과 통제, 테스트 분석과 설계, 테스트 구현과 실행, 완료 조건의 평가와 리프팅, 테스트 마감 활동이 있다.


테스트 설계

테스트 케이스를 도출하고 수행하여 테스트 대상이 어느 수준까지 테스팅이 되었는지 보장하기 위해 사용된다.


테스트 케이스 설계 기법

  • 명세 기반 기법
    • 동등 분할 : 대표값으로 수행하도록 테스트 케이스를 설계
    • 경계값 분석 : 경계 ㅂ분에 해당하는 입력값에서 결함이 발견된 확률이 높기 때문에 경계값까지 포함하여 테스트 케이스를 설계
    • 페어와이즈 조합 테스팅 : 테스트에 필요한 각 값들이 최소한 한번씩은 조합을 이루게 하는 기법
    • 결정테이블 테스팅 : 결정 테이블에 표시된 입력값과 자극의 조합을 테스트하도록 설계하는 기법
    • 상태전이 테스팅 : 상태전이도를 기반으로 상태전이 사이의 관계를 설계하는 기법
    • 유스케이스 테스팅 : 유스케이스에서 테스트 케이스를 도출하는 기법
  • 구조 기반 기법
    • 제어흐름 테스팅 : 모든 가능한 이벤트 흐름 구조를 테스트할 수 있게 설계하는 기법
    • 커버리지 테스팅 : 테스트 스위트에 의해 테스트되 정도인 커버리지를 달성하기 위해 테스트 케이스를 설계하는 기법
    • 최소비교 테스팅 : 입력값의 조합을 테스트하도록 테스트 케이스를 도출하는 테스트 설계 기법
  • 경험 기반 기법
    • 탐색적 테스팅 : 테스트를 수행하는 동안 얻은 정보를 활용하는 비공식적인 테스틑 설계 기법
    • 분류 트리 기법 : 분류 트리로 표현된 테스트 케이스를 입력 및 출력 도메인의 대표값을 조합하여 수행

테스팅 유형

여러 유형이 존재하고 각 유형에서 테스트를 수행하는 주체, 테스팅 목적, 테스트 환경 등을 고려하는 것이 원칙

인수 테스트, 시스템 테스트, 통합 테스트, 단위 테스트가 있다.


테스팅 기법

  • 화이트 박스 테스팅 : 구조적 또는 코드기반 테스트이며 단위 테스팅에서 주로 사용 되고 테스트가 가능한 소프트웨어 내에서 결함을 찾고 해당 기능을 검증
  • 블랙박스 테스팅 : 기능 또는 명세기반 테스트이며 시스템 테스팅에서 사용되고 기능 및 비기능 요구사항을 모두 검증

리팩토링이란?

외부에서 보는 프로그램 동작은 변경하지 않고 프로그램 내부 구조를 개선하는 것


리팩토링 수행 절차

  • 조직 구성 : 멘토 역할의 선임자와 복합 구성
  • 수행 통제 : 변경 관리, 형성관리 수행, CCB 통제
  • 수행 기법 : 디자인 패턴, AOP 수행
  • 테스트 : 단위 / 통합 테스트, 회귀 테스트
  • 결과 정리 : 문서화, 현행화, 운용 시스템 적용

코드스멜이란?

일기 어렵거나 중복된 로직을 가진 프로그램처럼 개발자가 이해하거나 유지보수하기 어려워 리팩토링의 대상이 되는 코드


대표적인 리팩토링 기법

  • 메소드 정리 : Extract Method, Replace Parameter with Method
  • 객체 간 기능 이동 : Extract Class, Extract Subclas, ExtractInterface
  • 이름 : Rename Method
  • 추측성 일반화 : Inline Method, Collapse Hierarchy
  • 중복 : Replace Magic Number with Symbolic Constant, Pull Up Field, Pull Up Method

데이터와 데이터베이스의 이해

데이터, 정보, 지식에 대한 개념과 특징

  • 데이터 : 현실세계에서 발견, 조사, 수집, 창작을 통한 기초자료인 수집 자원의 형태 그대로의 것
  • 정보 : 다양한 데이터를 목적에 맞게 일정한 규칙에 따라 정리하고 분류하여 체계화한 형태
  • 지식 : 수많은 구체화된 정보로부터 일반화된 사항

데이터 처리 유형의 개념과 특징

  • 일괄처리시스템 : 데이터를 일정기간 또는 일정량을 모아서 한꺼번에 처리하는 방식
  • 온라인처리시스템 : 데이터가 컴퓨터로 전송되면 컴퓨터는 즉시 데이터를 처리하는 방식
  • 분산처리시스템 : 지리적으로 분산되어 있는 처리기와 데이터베이스를 네트워크로 연결하여 처리하는 방식

파일처리시스템

각 개별 응용프로그램이 자신이 처리할 개별적인 파일을 접근하여 검색, 입력, 삭제 및 수정을 하는 것
  • 응용 프로그램은 응용프로그래머가 생각하는 논리적 파일 구조를 직접 물리적 파일 구조로 구현해야 함
  • 응용 프로그래머는 물리적 데이터구조에 대해 잘 알고 있어야만 데이터에 대한 접근 방법을 응용 프로그램 속에 구현시킬 수 있음
  • 각 응용 프로그램마다 각자의 데이터 파일을 가지고 있는 환경하에서 데이터의 공용은 자연히 어렵게 되어 결국 하나의 파일은 하나의 응용만을 위해 존재

데이터베이스

통합되고, 운영되고, 저장되고, 공유되며 중복된 데이터를 한 군데 집약시켜서 최대한 중복을 배제한 상태에서 관리
  • 실시간 접근성 : 수시적이고 비정형적인 질의에 대하여 실시간 응답
  • 계속적인 변화 : 갱신, 삽입, 삭제
  • 동시 공용 : 동일 데이터를 여러 사람이 다른 방법으로 동시에 공용할 수 있도록 자원
  • 내용에 의한 참조 : 위치나 주소가 아닌, 사용자가 요구하는 데이터의 내용, 즉 값에 따라 참조

데이터베이스 시스템(DBS)

데이터를 Database에 저장하고 관리해서 필요한 정보를 생성하는 컴퓨터 중심의 시스템
  • 데이터베이스 : 한 조직의 여러 응용시스템이 공용하기 위해 최소의 중복을 통합, 저장된 운영 데이터의 집합
  • 데이터베이스 언어 : 사람과 시스템의 인터페이스를 제공하는 도구
  • 사용자 : 데이터베이스 관리자, 데이터베이스 응용 프로그래머, 데이터베이스 사용자
  • 데이터베이스 관리시스템 : 데이터베이스를 구축하고 이용하는 기능을 제공하는 시스템 소프트웨어

데이터독립성

지속적으로 증가하는 유지보수 비용을 절감하고 데이터 복잡도를 낮추며 중복된 데이터를 줄이고 DB에 대한 사용자의 View와 DB가 실제 표현되는 View를 분리하여 변경 간섭을 주이는 것이 핵심 목적


ANSI-SPARC의 3-Level Database Architecture

  • 외부스키마 : 개개 사용자 단계로서 개개 사용자가 보는 개인적 DB 스키마
  • 개념스키마 : 모든 사용자 관점을 통합한 조직 전체의 DB를 기술하는 것
  • 내부스키마 : 물리적 장치에서 Data가 실제적으로 저장되는 방법을 표현하는 스키마

데이터베이스 관리자

데이터베이스 구성 및 관리 운영 전반에 대한 책임을 가지는 역할
  • 데이터 모델링
  • DB 물리설계
  • 튜닝(성능개선)
  • DB 구축
  • DB 운영
  • DB 표준화

데이터아키텍트

정책 및 기준을 수립하여 모델링하고 체계화하는 역할 수행자
  • 데이터 관리체계 수립
  • 데이터 표준수립
  • 데이터 모델링 수행
  • 데이터 보안체계 수립

Database Management System(DBMS)

응용 프로그램들이 데이터베이스를 공용할 수 있게 관리해 주는 소프트웨어 시스템
  • DDL 컴파일러 : DDL로 명세된 스키마를 내부 형태 즉, 메타데이터로 처리하여 시스템 카탈로그에 저장
  • 질의어 처리기 : 일반 사용자가 제출한 고급 질의문을 검사하고 파싱해서 컴파일한다.
  • DML 예비 컴파일러 : 응용 프로그램에 삽입된 DML명령문을 추출하고, 추출된 DML 명령문은 데이터베이스 접근을 위한 목적코드로 컴파일 되도록 DML컴파일러에 보내짐
  • DML 컴파일러 : 넘겨받은 DML 명령어를 파싱하고 컴파일하여 목적코드를 생성함
  • 런타임 데이터베이스 처리기 : 실행시간에 데이터베이스 접근을 관리
  • 트랜잭션 관리자 : 데이터베이스에 접근하는 과정하는 과정에서 무결성 제약조건의 만족여부, 사용자의 권한 검사 등 체크
  • 저장 데이터 관리자 : 디스크에 저장되어 있는 사용자 데이터베이스나 카탈로그 접근을 책임짐

데이터베이스 설계 및 구축절차

데이터베이스 설계 및 구축과정

  1. 요구사항 수집 및 분석 : 사용자가 요구하는 업무 요구사항을 수집하고 분석한다.
  2. DB 설계
    • 개념적 데이터 모델링 : 현실 세계에서 나타나는 정보 구조를 추상적으로 개념화
    • 논리적 데이터 모델링 : 개념적 설계의 결과를 데이터베이스 저장이 용이한 논리적 구조로 변환
    • 물리적 데이터 모델링 : 데이터베이스의 물리적 저장 구조 결정
  3. DB 구축 : DB 구성에 필요한 SQL 문을 작성하고 이를 실행하여 사용자 요구사항을 수용할 수 있는 데이터베이스를 생성
  4. 운영 및 유지보수

데이터베이스 설계 고려사항

  • 분석대상 산출물
    • 개념 모델 설계 단계 고려사항 : 주제영역, 핵심 엔터티, 관계가 정의되어 개념 데이터 모델이 산출물로 작성
    • 논리 모델 설계 단계 고려사항 : 최종 개체-관계 다이어그램이 작성
    • 물리 모델 설계 단계 고려사항 : 데이터베이스가 구현되기 위한 물리적인 스키마를 구현
  • 설계 주체 고려사항
    물리 모델링 단계에서는 효율적이고 안정적인 성능을 위해 데이터 아키텍트와 데이터베이스 관리자가 각자의 관점에서 최선이라고 판단되는 모델을 상호 협의하여 최종 물리 모델을 결정

프로젝트 라이프사이클에서 데이터베이스 설계 영향도

  • 정보공학이나 구조적 모형에 기반한 방법론에서는 보통 분석단계에서 업무중심의 논리적인 데이터 모델링을 수행하고 설계단계에서 하드웨어와 성능을 고려한 물리적인 데이터 모델링
  • 나선형 모델에서는 업무크기에 따라 논리적 데이터 모델과 물리적 데이터 모델이 분석설계단계에 양쪽에서 수행

데이터 측과 어플리케이션 축으로 구분이 되어 프로젝트가 진행되면서 각각에 도출된 사항은 상호 검증을 지속적으로 수행하면서 단계별 완성도를 높이게 된다.
