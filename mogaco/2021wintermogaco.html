Client Side Basic

Cross Site Scripting (XSS)

서버의 응답에 공격자가 삽입한 악성 스크립트가 포함되어 사용자의 웹 브라우저에서 해당 스크립트가 실행되는 취약점

  • Stored XSS : 악성 스크립트가 서버 내에 존재하는 데이터베이스 또는 파일 등의 형태로 저장되어 있다가 사용자가 조회하는 순간 발생
  • Reflected XSS : 사용자의 요청 데이터가 서버의 응답에 포함되는 과정에서 공격 스크립트가 그대로 출력되어 발생

Mitigations

  • Server-side Mitigations : 사용자의 입력을 화이트리스트 필터링함
  • HTTPOnly Flag : 서버에서 응답 헤더에 Set-Cookie 헤더를 전송해 자바스크립트에서 해당 쿠키에 접근하는 것을 금지함
  • Content Security Policy : 지시어를 적용하여 사이트에서 로드하는 리소스들의 출처를 제한
  • X-XSS-Protection Header : XSS Filter를 활성화하여 전송된 Request 값이 XSS 공격 발생을 판단하여 사용자에게 알리고 차단함

Cross Site Request Forgery

비정상적으로 사용자의 의도와 무관하게 다른 사이트에 HTTP 요청을 보내는 것

src와 href에 주소와 세션쿠키의 값을 설정하여 비정상적인 요청을 보내면 취약점이 발생

ex) <img src="/sendmoney?to=dreamhack&amount=1000">

Mitigations

  • 세션 쿠키 대신 커스텀 헤더를 사용하여 사용자 인증
  • 공격자가 예측할 수 없는 파라미터 추가 및 검증

Open Redirect

Redirect는 사용자의 Location을 이동시키기 위해 사용하는 기능이며 이동하는 주소가 공격자에 의해 변조될 경우 Open Redirect 취약점이 발생함

Mitigations

  • 이동을 허용한 주소에 대해서만 이동하게 함

Click Jacking

화면 출력에 영향을 미치는 요소들을 이용하여 사용자의 눈을 속여 사용자의 클릭을 유도하는 공격방법

Mitigations

  • 부모 페이지의 URL을 제한하는 방식으로 방지, ex) X-Frame-Options, frame-ancestors

  • Mogaco

Server Side Basic

Injection

사용자의 입력 값이 어플리케이션의 처리 과정에서 구조나 문법적인 데이터로 해석돼 발생하는 취약점

  • SQL Injection : SQL을 사용할 때 공격자의 입력값이 정상적인 요청에 영향을 주는 취약점
  • Command Injection : OS Command를 사용 시 사용자의 입력 데이터에 의해 실행되는 Command를 변조할 수 있는 취약점
  • Server Side Template Injection : 템플릿 변환 도중 사용자의 입력 데이터가 템플릿으로 사용돼 발생하는 취약점
  • Path Traversal : URL / File Path를 사용 시 사용자의 입력 데이터에 의해 임의의 경로에 접근하는 취약점
  • Server Side Request Forgery : 공격자가 서버에서 변조된 요청을 보낼 수 있는 취약점

File Vulnerability

  • File Upload : 서버의 파일 시스템에 사용자가 원하는 경로 또는 파일 명 등으로 업로드가 가능하여 악영향을 미칠 수 있는 파일이 업로드하는 취약점
  • File Download : 서버의 기능 구현 상 의도하지 않은 파일을 다운로드할 수 있는 취약점

Business Logic Vulnerability

  • Business Logic Vulnerability : 정상적인 흐름에서 검증 과정의 부재 및 미흡으로 인해 정상적인 흐름이 악용되는 취약점
  • Inseure Direct Object Reference(IDOR) : 변조된 파라미터 값이 다른 사용자의 오브젝트 값을 참조할 때 발생하는 취약점
  • Race Condition : 비즈니스 로직의 순서가 잘못되거나, 한 오브젝트에 여러 요청이 동시에 처리되는 사오항에서 발생하는 취약점

Language specific Vulnerability

언어에서 제공하는 모든 함수가 안전한 것은 아니다. 의도한 기능을 제공하는 함수를 잘못 사용해서 문제가 발생할 수도 있고 사용자의 입력 데이터가 함수의 인자로 사용되어 취약점이 발생함

  • Common

    • Eval : 인자로 입력된 문자열을 어플리케이션 코드로 실행
    • OS command function
    • Filesystem function
    • Serialize / Deserialize : 오브젝트, 데이터를 특정 타입으로 변환하는 것이 직렬화

  • PHP

    • Include : 인자로 전달된 파일을 읽은 후 해당 파일의 내용을 출력
    • Wrapper : 파일 시스템 함수에서 URL style 프로토콜을 위한 wrapper들이 존재
    • Extract : 배열에서 변수를 가져옴, 변수의 데이터를 덮을 수 있음
    • Type Juggling : 자동 형변환으로 인해 의도치 않은 결과 발생
    • Session : 세션에 사용자의 입력 데이터를 넣어서 다른 공격과 연계하여 사용
    • Upload Logic : php 코드상에서 업로드를 구현하지 않아도 임시 디렉토리에 업로드 가능

  • Javascript

    • Comparison Problem : js는 값을 비교할 때 valueOf, toString을 사용하여 원시 값을 비교하기 때문에 비교 후 연산에서 오류가 발생 가능
    • Prototype Pollution : 객체를 생성할 때 초기화 할 프로토타입을 지정하지 않으면 Object.prototype을 상속받음. 같은 프로토타입을 상속받은 객체는 같은 주소를 가르킴. 따라서 객체의 프로토타입을 오염시켜 취약점을 발생함

Misconfiguration

잘못된 설정으로 인해 공격자는 허가되지 않은 동작 수행가능하며 로직인 아닌 간단한 설정 오류로 인해 발생함

  • 부주의로 인해 발생하는 문제점 : 메뉴얼에 존재하지 않은 설정이거나 개발자의 부주의로 인해 설정 유무를 알지 못한 상태에서 서비스를 할경우 발생
  • 편의성을 위한 설정에 의해 발생하는 문제점 : 개발자를 위한 설정이 서비스 환경에서도 켜져있어 공격자가 이를 통해 시스템 내부의 정보를 알아내 추가적인 공격에 사용
  • 메뉴얼과 실제 구현체의 차이로 인해 발생하는 문제점 : 메뉴얼에 설명되어있는 대로 사용했지만 중의적 표현이 존재하거나 잘못 설명되어 있을 때 발생
  • 해당 코드나 설정에 대한 이해 없이 사용해 발생하는 문제점 : 취약점이 존재하는 예제코드, StackOverflow 답변등을 이해 없이 복사 붙여넣기 해 사용하거나 메뉴얼에 존재하는 권고 설정을 무시한 채 사용해 발생
